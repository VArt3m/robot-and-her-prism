# CLAUDE.md — robot_prism_html

Browser port of a 2-D light-beam puzzle. Pure ES modules, **no build step, no framework, no deps**. See `README.md` for gameplay/controls/features; this file is for navigating the code.

> The active, canonical implementation lives here. The sibling `../robot_prism_py/` is legacy — do not use it (see the parent `../CLAUDE.md`).

## Layout (dependency flow is one-way: core → sim → ui)

```
js/core/   constants.js (physics consts, COLORS, WORLD_W/H)  entities.js (Node/Wall/Barrier/ForceField)
           geometry.js (seg_inter, pt_seg_dist, dist…)  strings.js (STR, i18n; incl. STR.panel)
js/sim/    engine.js   light/logic/forcefield solver  (solve + step, ~700 ln); ray_clear (shared LOS)
           motion.js   player/box movement, reach, placement_spot
           world.js    scene state + FACADE over Engine/Motion (read-through getters)
           objects.js  OBJECT_TYPES table (material/carriable/requiresTarget/programmable/jammable/radius)
           targeting.js   TARGET_SPECS — per-kind targeting data (incl. candidates()) behind a shared core
           programming.js PROGRAM_SPECS — per-kind programming data (FACETS) behind a shared core
           relays.js   RELAY_SPECS — connectors/inverters/mixers/sisters: per-kind cleanEmit + shared relayEmit (corruption)
           colormix.js additive colour model (RGB-bit OR): red/green/blue, yellow/magenta/cyan, white
           occlusion.js   RAY_OCCLUSION + carriedRayType + visibilityPolygon (the "passable ray area")
           level.js    the built-in levels: build_world(scratch) + the LEVELS registry (build_level() = Test Grounds, the default)
js/ui/     renderer2d.js  Canvas2D (DPR-aware, letterboxed); render(world, uiState); world↔screen transforms
           input.js       keyboard/mouse → events + held/drainActions (Q/F highlight holds live in `held`)
           panel.js       DOM overlay tool panel (top-left): highlight toggles, reset-hold, undo, level-select-hold
           app.js         game loop, all play interactions (~1500 ln, the big one)
```

## Key patterns — learn these before adding features

- **Spec-driven frameworks.** Targeting, programming, and relays are each ONE generic core driven by per-kind data. To add a targetable/programmable device: set the flag in `OBJECT_TYPES` (`objects.js`) and add one spec entry — **do not add per-kind branches in `app.js`**. The frameworks' own doc-comments explain every spec field. `test_targeting.mjs` / `test_programming.mjs` enforce that every flagged kind has a conforming spec.
- **Relays (connector / inverter / future "sisters").** A relay is a field node that receives beams and RE-EMITS. They share ALL machinery — occlusion endpoint, the solver's propagate loops, the dead/confused map, links, buttons, carrying, box-stacking/elevation, recolour targeting — keyed off `isRelay(kind)` / `isRelayKind(kind)`, never `kind === 'connector'`. The ONLY per-kind difference is `cleanEmit(node, incoming)` in `relays.js` (connector passes one colour through; **inverter** has two programmable forms in `node.mode`: `'swap'` (default) swaps within `node.pair`, dark on none/conflict/off-pair — and `'complement'` ignores the pair and emits what is MISSING to make white from the OR of all incoming base colours (`7 & ~m`): a duo of primaries→the third primary, a full white collected at the input (three primaries / two colours where one is a secondary / white itself)→a real BLACK ray (the additive negative of white; black is a carriable colour with mask 0 — distinct from `null`/no-light), a black ray RECEIVED→white, a lone single base colour is too little→confused, and **no input→dark** (truly nothing in emits `null`/no beam; black only appears downstream of a real white, never from nothing) (no-input is NOT "received black"; black is a real colour awaiting a real black/environment source, so an unfed complement inverter never invents white from nothing — which also keeps it from radiating without a source); the `pair` programming facet is hidden while in `'complement'` form via its `available(node)` predicate, so to re-pick the pair you switch back to `'swap'` first; **mixer** COMBINES inputs into their additive OR and emits that: two primaries→their secondary, a lone secondary→itself (passes through), secondary+in-mix primary→that secondary, and any inputs covering all three primaries→**white** (a mixer can EMIT white); it is confused by a single base colour (the sum is one primary, nothing to mix) and by **white on the INPUT** (white is emit-only, never received). The mixer is a SINGLE, non-programmable form (no `node.mode`) and is NOT corruptible — `corruptible:false` on its `RELAY_SPECS` entry, surfaced as `isCorruptibleKind('mixer')===false`, so a Forge can't program it and a rewirer can't lock a colour onto it (its targeting list excludes non-corruptible relays), and `relayEmit`/`relayConfused` ignore any stray `color` on it). Colour is the additive model in `sim/colormix.js`: primaries are RGB bits, mixing is bitwise OR, giving seven colours — red/green/blue, yellow/magenta/cyan, white — plus BLACK (mask 0), a real carriable colour that is NOT the same as `null` (no light): `maskColor(0)` stays `null` so generic mixing never invents black from an empty sum; black is only emitted where meant (the complement inverter from a full white). Corruption (`node.color`, null = clean) is shared by `relayEmit`: a corrupted relay emits its locked colour on any input and never dies on conflict — except a kind that opts out with `corruptible:false` (the mixer; `isCorruptibleKind`), whose `color` is ignored everywhere. Whether a dark output is an ERROR (the dotted dead/confused mark) or a DELIBERATE dark is decided by `relayConfused` (engine `_conflict_map` uses it): the default is "any clean relay dark-while-lit is confused", but a spec may define `isConfused` to override — the complement inverter is confused ONLY on a lone primary, never when it makes black from a full white. To add a sister: add an `OBJECT_TYPES` entry with `relay:true`, one `RELAY_SPECS.cleanEmit`, a `TARGET_SPECS` wiring spec (reuse `WIRE_SPEC()`), a `PROGRAM_SPECS` entry **if it is programmable** (the mixer has none), and strings (kind/glyph/tooltip). `test_inverter.mjs` / `test_mixer.mjs` are the models. (A mixer auto-ranks DOWNSTREAM of its feeders — it consumes feeders strongest-tier first and ranks at 1 + the tier that first lets it emit, so a near pair can satisfy it before a more distant third primary does — so NO manual placement is needed; while confused it accepts food from any direction until it completes, and if it can never complete it stays unranked and dark; see the directional bullet.)
- **Programming specs are FACETS.** Each `PROGRAM_SPECS[kind]` is `{ facets:[…] }`; most devices have one facet, but a device may expose several (the inverter has a corruption facet + a `pair` facet). `programOptions(spec, node, {canCorrupt})` flattens facets, drops no-ops, and gates corruptions at a clean-only Forge; each option is self-describing (its own field/labels/liveness). `stampProgramDefaults` stamps every facet default on pickup. `sameProgramValue` treats arrays as unordered sets (colour pairs).
- **Accumulator (`accumulator`) — a PORTABLE source with a charge.** Not a relay; `isAccumulatorKind(kind)` is the source of truth, `emitsAsSource(n)` (engine) = a source OR a charged accumulator. State is its `color` field. CHARGED (colour set): seeded at rank 0 by `_sourceRanks` and emit-init (both propagate paths), it emits ONLY its own colour — it never relays incoming light and is never in the dead/confused map (it is not a relay), and it is dead weight to beams passing through (a material non-relay blocker, owner=id, via the existing `_leveled_blockers` else-branch). Against a same-rank emitter (source / charged accumulator) any meeting tugs at the midpoint and delivers to neither end — a different colour clashes (two-tone split), the same colour is drawn as one connected ray but still passes nothing (the generalized cap above). EMPTY (null): a sink that charges over `ACCUM_FILL_SEC` of an uninterrupted SINGLE delivered colour (`_update_accum_charge`, real-time in `step` only; a conflict or interruption resets it; a cold `solve` clears it); `ready_accumulators()` surfaces completion and the UI's `_commitFills` (mirrors `_commitRecolors`) stamps the colour and drops the filling link (undoable — `color`+`links` are snapshotted). Wired like a connector via `toggle_link` (broadened to allow accumulator endpoints) and a state-aware `TARGET_SPECS.accumulator` (empty: sources/relays; charged: + receivers). DISCHARGE is a carried-only action: the `Q` key (`'discharge'`) and the panel's hidden Discharge button (revealed by `canDischarge`), both silent unless a charged accumulator is in hand. `test_accumulator.mjs` is the reference.
- **Light is directional by FEEDER rank.** Links are undirected, but the solver gives light a direction so it cannot loop back upstream (which used to confuse an inverter wired into another relay). Rank is no longer a topological link-distance — it is computed from what actually ARRIVES, co-evolving with `emit` inside the `propagate` fixpoint (not precomputed): sources and charged accumulators are rank 0; for each colour a relay receives, keep its STRONGEST (lowest-rank) feeder, then feed it those feeders by TIER, strongest first — take all rank-0 feeders at once and check whether it can emit, else add all rank-1 feeders, and so on, STOPPING at the first tier that lets it emit: it consumes exactly that prefix and ranks at **1 + that tier** (`_gatherIncoming` → `_relayIntake`), sitting strictly below every colour it consumes and leaving weaker feeders out. If no tier ever lets it emit, it is genuinely confused — unranked (+∞) and dark, radiating nothing. An unfed relay is unranked (+∞). `engine._linkRanks()` is now a back-compat shim returning the converged `propagate()` ranks. In `_beams_and_cross`, a higher-rank→lower-rank beam is **suppressed when the connection is complete** (nothing material between them, i.e. `hard_mat` reaches); with a wall between, the connection is incomplete and BOTH ends radiate at it; unreached pairs are never suppressed. **Same-rank** linked relays are peers — neither feeds the other: on a clear path each beam is **capped at the midpoint** (per-beam in `_beams_and_cross`, NOT a mutual cut that needs both ends lit — so it stays stable even when a node consumes a same-rank relay, e.g. a mixer) and delivers to neither end, **regardless of colour**. The `tug` tag only drives the renderer: a **same-colour** tug (`'same'`) is drawn as one smooth full-length ray (a visual link that still delivers nothing), a **different-colour** clash (`'clash'`) as a two-tone split, and a same-rank relay that is dark yet NOT confused (not seeking food) soft-caps (`'soft'`) — shown full length, feeding neither. A pure sink — a receiver, or an EMPTY accumulator charging up — is no peer, so the beam runs full length and delivers (a source must light a receiver / fill an accumulator). **CONFUSED relays don't push back — they look for food first.** A relay that took light but cannot yet make a valid output (`relayConfused`) holds only a PROVISIONAL rank, so while confused it accepts incoming light from EVERY direction: BOTH the same-rank cap AND the higher→lower suppression are lifted for a confused target (`!confused.has(t)` guards both). This is what lets a combiner auto-place downstream of its feeders and stops it starving when a colour only arrives via a longer (higher-rank) chain — e.g. red direct@0 + green via a 2-hop chain@2 makes the mixer climb to rank 3 and complete to yellow instead of rejecting the green@2. A confused relay emits `null` (pushes nothing back), and once completed it re-ranks strictly above its feeders and suppresses the back-direction toward them, so the fixpoint can't oscillate. `test_directional.mjs` and `test_rank_feeders.mjs` are the references.
- **Programming is gated behind Forges.** Targeting a carried device is just a **click on a target** while carrying it (`_playClick` → `_applyTargetIntent`; a click on the device's own intent ray erases it; `C` clears all). There is no separate click-by-click mode. The annulus press-and-hold (`_updateGestures` → `_startAimArrow`) only draws the **golden-arrow wiring** — held until release, no in-ring decision and no E-hold variant. Programming is summoned ONLY at a **Forge**: `app._invokeForge()` (the `F` key / the panel's hidden "Program" button) opens the programming chooser when a programmable item is carried within a Forge's `FORGE_REACH` and the Forge has `uses` left; each *applied* value spends one use (Escape closes for free). The chooser hides any value that wouldn't change the device (`programIsNoop`). A Forge is a stationary, non-carriable `forge` node that is **material** — it occludes light (`engine._leveled_blockers`) and blocks movement/placement (`motion.move_player`, `world.object_pos_blocked`) like other objects, via `world.forges()`. Connectors are `programmable` too (field `color`, `null` = clean): a Forge cleans or corrupts them, the same locked-colour state a fired rewirer leaves. Two Forge rules to know: (1) a Forge has a hidden `corrupts` flag (default true); a **clean-only** Forge (`corrupts:false`, drawn with a cool steel rim instead of warm brass) can strip corruption but offers no corruption values — `isCorruptionValue(spec, value)` + the spec's `corrupts`/`cleanValue` decide which menu values are corruptions. (2) Programming needs the robot in EXACTLY ONE Forge's area; sitting in the **overlap** of two control areas (or out of all of them) makes `_invokeForge` a silent no-op and `_canForge` false. The renderer's Forge control-zone overlay (drawn before the nodes, when carrying a programmable item) shows the reach of only the Forge(s) the robot is **standing in** — one zone in a single area, both when in their shared exclusion lens (an invitation to step out and disambiguate) — each carved against every live Forge so any dead overlap stays an unfilled hole rather than usable area.
- **World is a facade.** `world.js` owns raw state (nodes, boxes, links, jam_links, recolor_links) and exposes solver outputs as read-through getters (`w.emit`, `w.lit`, `w.beams_draw`…) that delegate to `engine`. Movement delegates to `motion`. Add new derived state to the engine, not the world.
- **Solve vs step.** `world.solve(cold, fillInstant)` = full recompute (call after any structural/state change or restore). `world.step(now)` = per-frame advance of animated charge/beams; returns "still live". `app._tick` only steps when something is dirty/live.
- **Undo = snapshot inputs only.** `app._captureState` / `_restoreState` snapshot *input* state (positions, links, per-node color/fuse/spent); derived state is recomputed by `solve()` on restore. `_sig()` is the cheap signature used to decide if a drag actually changed anything worth an undo point. Max 3 steps (`UNDO_MAX`).
- **Carry "shadow".** While carrying, `_syncCarried` + `_resolvePlacement` + `motion.placement_spot` continuously compute a guaranteed-legal, in-reach drop spot, so `_commitPlacement` needs no further reach test. Stacking legality lives in `canPlace` (`app.js`).
- **Player collision is RADIUS-AWARE; the purple field DROPS carries on contact.** `motion._static_blocked` treats the robot as a DISC swept old→new and blocks when that sweep comes within `PLAYER_R-2` of any static blocker (`geometry.seg_seg_dist`), not merely when her centre line crosses one. This seals the off-by-a-pixel gaps where blocker segments almost meet — force-field ends vs. the `x=1006` border, the tan(`782`)/purple(`786`) jog at `y≈510`, the tan top vs. the `y=12` border, wall corners — which a centre-only crossing test let her slip her body through. It can't trap her: she spawns well clear and the check never lets the centre inside the skin, while sliding PARALLEL (distance stays at the radius, not below it) is always allowed. The carry-only purple field is unchanged as a blocker (`staticBlockerSegs` `barriers:'carry'` includes purple only when `carry`); `motion.blocked_by_purple(old,new)` = blocked-while-carrying ∧ clear-empty-handed isolates *that* obstruction, and `app._tick` uses it to AUTO-DROP a carried object the instant she walks it into the field — set down on the NEAR side (it never crosses; `reach_blocked` already forbids placing across the field) so she continues through empty-handed — EXCEPT when the cursor sits inside her activation ring (`dist(player,mouse) < CONNECT_REACH`), read as deliberate by-hand placement, which just blocks as before. `test_collision.mjs` pins the junction sealing, full traversability, the parallel-slide / head-on-stop behaviour, the purple isolation, and both auto-drop branches.
- **Self-occlusion is dwell-gated and must stay STABLE.** Standing in a beam for `DWELL_THRESH` flips `world.player_block` on (via `app._tick`), making the robot a material blocker that cuts the beam at her body — so she can walk through rays but blocks one she lingers in. The trigger, `engine.player_on_beam()`, judges "on a beam" against each beam's **player-INDEPENDENT** length (`hard_mat`, which excludes the player; full length for always-full tug/soft beams) — NEVER against the drawn/player-cut beam. This is load-bearing: once the block engages it retracts the drawn beam to her near edge, and for an off-centre / diagonal hit that endpoint is >`BEAM_TOUCH` from her centre, so a test against the cut beam would instantly read "not touching", drop the block, and flicker (a historical, position-dependent blink). `test_player_block.mjs` freezes this invariant.
- **Rays occlude the directional law (static + timed).** The higher→lower suppression above asks whether the connection is *complete*; a live beam crossing the path now counts as wall-like for that question (`engine._rayCrosses`), so the back-beam is emitted and then cut at the crossing by the ordinary crossing mechanic — a ray genuinely IS a wall for deciding what reaches a hungry tool. The obstacle set is `_obstaclesFromBeams` (each beam spanning the length it actually reaches). On the **timed** `step` it is the PREVIOUS frame's beams (`_buildRayObstacles`), fixed for the frame. On the **cold** `solve` there is no previous frame, so `propagate()` (called with `selfObstacles`) threads the previous PASS's beams back in and iterates until the beam field is self-consistent (the loop re-runs while `_obstacleKey` keeps changing), capping like any fixpoint. Because the radiated beam can cut its own obstacle, a crossed configuration oscillates live with the `CUT_LINGER` (now **0.75 s**) lag and lands on one deterministic phase statically; that self-oscillation is INTENDED, not a bug to damp. The rule never adds the player as an obstacle, so it can't stop her passing through rays, and ordinary (non-crossing) configs never fire it — every existing cold-solve suite is unchanged. `test_ray_occlusion.mjs` pins the mechanism, the wiring (cold solve populates / step seeds `_ray_obstacles`), static termination/determinism, and that a straight chain never fires it.
- **Strings.** All user-facing text goes through `STR.*` (`core/strings.js`), never inline literals.
- **IDs/links.** Light links are a `Set` of sorted `"a\x00b"` keys; jam/recolor intents are `Map`s (one target per device, re-mark overwrites).
- **Tool panel + highlights.** `panel.js` is a DOM overlay (not canvas-drawn) built by `app.js`; it reports presses through callbacks and is refreshed each frame via `panel.refresh()`. Two highlights — "possible targets" and "passable ray area" — are each the XOR of a sticky panel toggle and a momentary held key (`Shift` → `held.has('hl_targets')`, `Space` → `held.has('hl_passable')`), so holding the key *suppresses* a toggled-on highlight. (Shift + Space sit under one palm and dodge the browser-owned Alt/Ctrl chords; `input.js` preventDefaults Space so it neither scrolls nor activates a focused button.) `app._updateHighlights()` computes `uiState.{showTargets,showPassable,targetHints,passableRegion}`; the renderer just draws them. **"Possible targets" is two-tier**: `targetHints` is populated WHENEVER a targeting device is carried (so a **faint dotted golden contour** always rings every target — `showTargets` does NOT gate its existence), and `showTargets` only raises the renderer to the *prominent* version (bright solid ring for a clear shot, grey dashed for a blocked one). It is spec-driven: each `TARGET_SPEC` exposes `candidates(world, deviceId, origin)` (reachability via `world.ray_clear`); add the hook, never per-kind UI branches. "Passable ray area" is `occlusion.visibilityPolygon` over the carried item's `carriedRayType`. The panel politely fades (`setConflict`, ~1 s) when it covers a live target the player is reaching for during the **golden-arrow link drag** — `app._panelConflict()` decides this from the panel footprint vs. candidate screen positions.
- **Reset/undo tuning.** `UNDO_MAX = 6`, `RESET_HOLD_SEC = 2` (in `app.js`), honoured identically for the keyboard (Z / hold R) and the panel buttons; the panel's reset hold is merged with the key in `_updateResetHold`.
- **Levels + the level-select overlay.** `sim/level.js` builds from one shared geography via `build_world(scratch)`: `scratch` ON is **Test Grounds** (the full sandbox — every scratch relay/gadget + both Forges); OFF is **Lorem's Puzzle #1** (the identical room MINUS both Forges and the central pickables, keeping one central connector `con_c` plus the two lower-right-room connectors `con_1`/`con_2`). The `LEVELS` registry (`[{id,name,build}]`) lists both, and registry ORDER is what the player sees: the **first entry loads at startup and sits first in the menu — now Lorem's Puzzle #1** (the App constructor builds `LEVELS[0]` to match `_levelId`). `build_level()` is no longer the startup level; it survives only as a back-compat helper still returning Test Grounds, used by the headless placement/highlight suites (and `test_integration`/`test_tree` now `_loadLevel('test_grounds')` after constructing the App, since they drive the full sandbox). The picker is a plain-click panel button (`data-act="levels"`, `.ui-levels` — a cautioning red) with **no keyboard shortcut** (panel-only; its `openLevels` callback calls `_openLevelMenu` directly). It opens a **modal** overlay (`uiState.levelMenu = { items:[{id,name,current,rect}], panel, titleY, footY }`, laid out in `_openLevelMenu`, drawn last in `renderer2d` over a dimmed playfield): `_tick` freezes the world while it is open (still draining only `escape`), `_onMouseDown` routes clicks to `_handleLevelMenuClick` (a hit loads via `_loadLevel`; a miss **closes** the overlay — a click outside cancels, like Escape), Escape closes it first in `_handleAction`, and the tool panel goes `inert` (`panel.setInert`). `_loadLevel(id)` rebuilds the world and clears transient play state (like a reset) and records `_levelId`; a full reset (`_fullReset`) now rebuilds the CURRENT level via `_buildCurrentLevel`, not always Test Grounds. `test_levels.mjs` (the registry + the two levels' contents) and `test_levelselect.mjs` (the overlay state machine, headless) are the references. (Geometry note: the upper-left wall runs `[235,13]→[235,146]` — its bottom was lengthened from `138` so a connector at the lower-right room's purple-field lip can no longer thread a beam just under it to `src_red`; the player doorway below, down to the lower wall at `y=490`, stays amply wide.)

## Run & test

- **Run:** `python server.py` → http://127.0.0.1:8080 (server forces correct JS MIME). Or `npm run serve` / `npx serve .`.
- **Tests:** 20 headless suites in `test/` covering beam physics, player collision + carry physics, the devices, the targeting/programming frameworks, the UI/app flow, and the levels. Each is a standalone script that prints its tally and exits non-zero on failure. **`TESTS.md` (project root) is a human-readable guide to what every suite checks and how the harness works** — read it before adding or changing tests.
  - **All:** `npm test` (or `node run-tests.mjs`).
  - **Selected:** `node run-tests.mjs jam tree` — runs suites whose filename contains any given substring.
  - **One, directly:** `node test/test_targeting.mjs`.
- **Note:** `package.json` declares `"type":"module"` so the `.js` source imports as ESM under Node. Without it, Node treats `.js` as CommonJS and every suite fails with *"is a CommonJS module"*. The browser is independent (`index.html` loads everything as `type="module"`). The project still has no third-party deps; `package.json` exists only for the module type and the test/serve scripts.

## Status / not-yet-done (per README)

Level editor is stubbed (`uiState.mode 'play'|'edit'`, switch wired but unimplemented). Mine countdown/explosion not implemented (fuse is programmable but inert). Connector cleaning/corruption now exists via the Forge (no standalone "uncolour" device). A `renderer3d.js` is anticipated — keep the `render(world, uiState)` contract renderer-agnostic.
