# CLAUDE.md — robot_prism_html

Browser port of a 2-D light-beam puzzle. Pure ES modules, **no build step, no framework, no deps**. See `README.md` for gameplay/controls/features; this file is for navigating the code.

> The active, canonical implementation lives here. The sibling `../robot_prism_py/` is legacy — do not use it (see the parent `../CLAUDE.md`).

## Layout (dependency flow is one-way: core → sim → ui)

```
js/core/   constants.js (physics consts, COLORS, WORLD_W/H)  entities.js (Node/Wall/Barrier/ForceField)
           geometry.js (seg_inter, pt_seg_dist, dist…)  strings.js (STR, i18n; incl. STR.panel)
js/sim/    engine.js   light/logic/forcefield solver  (solve + step, ~700 ln); ray_clear (shared LOS)
           motion.js   player/box movement, reach, placement_spot
           world.js    scene state + FACADE over Engine/Motion (read-through getters)
           objects.js  OBJECT_TYPES table (material/carriable/requiresTarget/programmable/jammable/radius)
           targeting.js   TARGET_SPECS — per-kind targeting data (incl. candidates()) behind a shared core
           programming.js PROGRAM_SPECS — per-kind programming data (FACETS) behind a shared core
           relays.js   RELAY_SPECS — connectors/inverters/mixers/sisters: per-kind cleanEmit + shared relayEmit (corruption)
           colormix.js additive colour model (RGB-bit OR): red/green/blue, yellow/magenta/cyan, white
           occlusion.js   RAY_OCCLUSION + carriedRayType + visibilityPolygon (the "passable ray area")
           level.js    the single hand-built level (build_level())
js/ui/     renderer2d.js  Canvas2D (DPR-aware, letterboxed); render(world, uiState); world↔screen transforms
           input.js       keyboard/mouse → events + held/drainActions (Q/F highlight holds live in `held`)
           panel.js       DOM overlay tool panel (top-left): highlight toggles, targeting, reset-hold, undo
           app.js         game loop, all play interactions (~1500 ln, the big one)
```

## Key patterns — learn these before adding features

- **Spec-driven frameworks.** Targeting, programming, and relays are each ONE generic core driven by per-kind data. To add a targetable/programmable device: set the flag in `OBJECT_TYPES` (`objects.js`) and add one spec entry — **do not add per-kind branches in `app.js`**. The frameworks' own doc-comments explain every spec field. `test_targeting.mjs` / `test_programming.mjs` enforce that every flagged kind has a conforming spec.
- **Relays (connector / inverter / future "sisters").** A relay is a field node that receives beams and RE-EMITS. They share ALL machinery — occlusion endpoint, the solver's propagate loops, the dead/confused map, links, buttons, carrying, box-stacking/elevation, recolour targeting — keyed off `isRelay(kind)` / `isRelayKind(kind)`, never `kind === 'connector'`. The ONLY per-kind difference is `cleanEmit(node, incoming)` in `relays.js` (connector passes one colour through; **inverter** has two programmable forms in `node.mode`: `'swap'` (default) swaps within `node.pair`, dark on none/conflict/off-pair — and `'complement'` ignores the pair and emits what is MISSING to make white from the OR of all incoming base colours (`7 & ~m`): a duo of primaries→the third primary, a full white collected at the input (three primaries / two colours where one is a secondary / white itself)→black which has no transmittable colour yet so reads as dark BUT is a deliberate output, NOT confusion, a lone single base colour is too little→confused, and **no input→dark** (no-input is NOT "received black"; black is a real colour awaiting a real black/environment source, so an unfed complement inverter never invents white from nothing — which also keeps it from radiating without a source); the `pair` programming facet is hidden while in `'complement'` form via its `available(node)` predicate, so to re-pick the pair you switch back to `'swap'` first; **mixer** COMBINES inputs — a single primary always confuses it — with two programmable forms in `node.mode`: `'blend'` sums (ORs) all inputs and emits the result iff it is a secondary (exactly two primaries present — so two primaries→their secondary, a lone secondary→itself, secondary+in-mix primary→that secondary; confused on one primary, white, or any sum reaching white), `'whiten'` emits white when the inputs together cover all three primaries, else dark). Colour is the additive model in `sim/colormix.js`: primaries are RGB bits, mixing is bitwise OR, giving seven colours — red/green/blue, yellow/magenta/cyan, white. Corruption (`node.color`, null = clean) is shared by `relayEmit`: a corrupted relay emits its locked colour on any input and never dies on conflict. Whether a dark output is an ERROR (the dotted dead/confused mark) or a DELIBERATE dark is decided by `relayConfused` (engine `_conflict_map` uses it): the default is "any clean relay dark-while-lit is confused", but a spec may define `isConfused` to override — the complement inverter is confused ONLY on a lone primary, never when it makes black from a full white. To add a sister: add an `OBJECT_TYPES` entry with `relay:true`, one `RELAY_SPECS.cleanEmit`, a `TARGET_SPECS` wiring spec (reuse `WIRE_SPEC()`), a `PROGRAM_SPECS` entry, and strings (kind/glyph/tooltip). `test_inverter.mjs` / `test_mixer.mjs` are the models. (A mixer that consumes another relay must sit DOWNSTREAM of it — a strictly higher rank — or the two are same-rank peers that split rather than feed; see the directional bullet.)
- **Programming specs are FACETS.** Each `PROGRAM_SPECS[kind]` is `{ facets:[…] }`; most devices have one facet, but a device may expose several written from one chooser (the inverter has a corruption facet + a `pair` facet; the mixer a corruption facet + a `mode` facet). `programOptions(spec, node, {canCorrupt})` flattens facets, drops no-ops, and gates corruptions at a clean-only Forge; each option is self-describing (its own field/labels/liveness). `stampProgramDefaults` stamps every facet default on pickup. `sameProgramValue` treats arrays as unordered sets (colour pairs).
- **Accumulator (`accumulator`) — a PORTABLE source with a charge.** Not a relay; `isAccumulatorKind(kind)` is the source of truth, `emitsAsSource(n)` (engine) = a source OR a charged accumulator. State is its `color` field. CHARGED (colour set): seeded at rank 0 in `_linkRanks` and emit-init (both propagate paths), it emits ONLY its own colour — it never relays incoming light and is never in the dead/confused map (it is not a relay), and it is dead weight to beams passing through (a material non-relay blocker, owner=id, via the existing `_leveled_blockers` else-branch). Against a same-rank emitter (source / charged accumulator) a colour clash tugs at the midpoint, same colour delivers (the generalized cap above). EMPTY (null): a sink that charges over `ACCUM_FILL_SEC` of an uninterrupted SINGLE delivered colour (`_update_accum_charge`, real-time in `step` only; a conflict or interruption resets it; a cold `solve` clears it); `ready_accumulators()` surfaces completion and the UI's `_commitFills` (mirrors `_commitRecolors`) stamps the colour and drops the filling link (undoable — `color`+`links` are snapshotted). Wired like a connector via `toggle_link` (broadened to allow accumulator endpoints) and a state-aware `TARGET_SPECS.accumulator` (empty: sources/relays; charged: + receivers). DISCHARGE is a carried-only action: the `Q` key (`'discharge'`) and the panel's hidden Discharge button (revealed by `canDischarge`), both silent unless a charged accumulator is in hand. `test_accumulator.mjs` is the reference.
- **Light is directional by rank.** Links are undirected, but the solver gives light a direction so it cannot loop back upstream (which used to confuse an inverter wired into another relay). `engine._linkRanks()` BFS-numbers emitting nodes by shortest link-distance from a source (source 0, its relays 1, theirs 2, …; receivers/carried nodes don't forward). In `_beams_and_cross`, a higher-rank→lower-rank beam is **suppressed when the connection is complete** (nothing material between them, i.e. `hard_mat` reaches). With a wall between, the connection is incomplete and BOTH ends radiate at it; same-rank/unreached pairs are never suppressed. This is purely topological + geometric (independent of the player and crossings), so it can't oscillate. **Same-rank** linked relays are peers — neither feeds the other. On a clear path each beam is **capped at the midpoint** (per-beam in `_beams_and_cross`, NOT a mutual cut that needs both ends lit — so it stays stable even when a node consumes a same-rank relay, e.g. a mixer), so the ray splits there and delivers to neither end and each relay keeps serving its other links. The one exception: if **both ends carry the same colour** there is nothing to clash over, so the beams run full length and the ray connects (delivers across) — a connector receiving its own colour is unchanged, so this stays stable too. The cap is a clash between two opposing emitters, so it only applies when the TARGET is a clash peer: it **emits a beam back** (`emit[t] != null` — a source, a charged accumulator, or a lit relay) OR **is a relay** (which could emit, and must still cap even while dark to block feedback). A pure sink — a receiver, or an EMPTY accumulator charging up — emits nothing back and is no peer, so the beam runs full length and delivers (a source must light a receiver / fill an accumulator). `test_directional.mjs` is the reference.
- **Programming is gated behind Forges.** Targeting is summoned by the annulus press-and-hold gesture / E-hold (`_updateGestures`, `_fireECarryTree` — TARGETING only now). Programming is summoned ONLY at a **Forge**: `app._invokeForge()` (the `F` key / the panel's hidden "Program" button) opens the programming chooser when a programmable item is carried within a Forge's `FORGE_REACH` and the Forge has `uses` left; each *applied* value spends one use (Escape closes for free). The chooser hides any value that wouldn't change the device (`programIsNoop`). A Forge is a stationary, non-carriable `forge` node that is **material** — it occludes light (`engine._leveled_blockers`) and blocks movement/placement (`motion.move_player`, `world.object_pos_blocked`) like other objects, via `world.forges()`. Connectors are `programmable` too (field `color`, `null` = clean): a Forge cleans or corrupts them, the same locked-colour state a fired rewirer leaves. Two Forge rules to know: (1) a Forge has a hidden `corrupts` flag (default true); a **clean-only** Forge (`corrupts:false`, drawn with a cool steel rim instead of warm brass) can strip corruption but offers no corruption values — `isCorruptionValue(spec, value)` + the spec's `corrupts`/`cleanValue` decide which menu values are corruptions. (2) Programming needs the robot in EXACTLY ONE Forge's area; sitting in the **overlap** of two control areas (or out of all of them) makes `_invokeForge` a silent no-op and `_canForge` false. The renderer's Forge control-zone overlay (drawn before the nodes, when carrying a programmable item) shows the reach of only the Forge(s) the robot is **standing in** — one zone in a single area, both when in their shared exclusion lens (an invitation to step out and disambiguate) — each carved against every live Forge so any dead overlap stays an unfilled hole rather than usable area.
- **World is a facade.** `world.js` owns raw state (nodes, boxes, links, jam_links, recolor_links) and exposes solver outputs as read-through getters (`w.emit`, `w.lit`, `w.beams_draw`…) that delegate to `engine`. Movement delegates to `motion`. Add new derived state to the engine, not the world.
- **Solve vs step.** `world.solve(cold, fillInstant)` = full recompute (call after any structural/state change or restore). `world.step(now)` = per-frame advance of animated charge/beams; returns "still live". `app._tick` only steps when something is dirty/live.
- **Undo = snapshot inputs only.** `app._captureState` / `_restoreState` snapshot *input* state (positions, links, per-node color/fuse/spent); derived state is recomputed by `solve()` on restore. `_sig()` is the cheap signature used to decide if a drag actually changed anything worth an undo point. Max 3 steps (`UNDO_MAX`).
- **Carry "shadow".** While carrying, `_syncCarried` + `_resolvePlacement` + `motion.placement_spot` continuously compute a guaranteed-legal, in-reach drop spot, so `_commitPlacement` needs no further reach test. Stacking legality lives in `STACK_RULES` (`app.js`).
- **Strings.** All user-facing text goes through `STR.*` (`core/strings.js`), never inline literals.
- **IDs/links.** Light links are a `Set` of sorted `"a\x00b"` keys; jam/recolor intents are `Map`s (one target per device, re-mark overwrites).
- **Tool panel + highlights.** `panel.js` is a DOM overlay (not canvas-drawn) built by `app.js`; it reports presses through callbacks and is refreshed each frame via `panel.refresh()`. Two highlights — "possible targets" and "passable ray area" — are each the XOR of a sticky panel toggle and a momentary held key (`Shift` → `held.has('hl_targets')`, `Space` → `held.has('hl_passable')`), so holding the key *suppresses* a toggled-on highlight. (Shift + Space sit under one palm and dodge the browser-owned Alt/Ctrl chords; `input.js` preventDefaults Space so it neither scrolls nor activates a focused button.) `app._updateHighlights()` computes `uiState.{showTargets,showPassable,targetHints,passableRegion}`; the renderer just draws them. "Possible targets" is spec-driven: each `TARGET_SPEC` exposes `candidates(world, deviceId, origin)` (reachability via `world.ray_clear`); add the hook, never per-kind UI branches. "Passable ray area" is `occlusion.visibilityPolygon` over the carried item's `carriedRayType`. The panel politely fades (`setConflict`, ~1 s) when it covers a live target the player is reaching for during a targeting gesture — `app._panelConflict()` decides this from the panel footprint vs. candidate screen positions.
- **Reset/undo tuning.** `UNDO_MAX = 6`, `RESET_HOLD_SEC = 2` (in `app.js`), honoured identically for the keyboard (Z / hold R) and the panel buttons; the panel's reset hold is merged with the key in `_updateResetHold`.

## Run & test

- **Run:** `python server.py` → http://127.0.0.1:8080 (server forces correct JS MIME). Or `npm run serve` / `npx serve .`.
- **Tests:** headless suites in `test/` (`test_integration`, `_jam`, `_occlusion`, `_placement`, `_programming`, `_rewirer`, `_targeting`, `_tree`, `_highlight`). Each is a standalone script that prints its tally and exits non-zero on failure.
  - **All:** `npm test` (or `node run-tests.mjs`).
  - **Selected:** `node run-tests.mjs jam tree` — runs suites whose filename contains any given substring.
  - **One, directly:** `node test/test_targeting.mjs`.
- **Note:** `package.json` declares `"type":"module"` so the `.js` source imports as ESM under Node. Without it, Node treats `.js` as CommonJS and every suite fails with *"is a CommonJS module"*. The browser is independent (`index.html` loads everything as `type="module"`). The project still has no third-party deps; `package.json` exists only for the module type and the test/serve scripts.
